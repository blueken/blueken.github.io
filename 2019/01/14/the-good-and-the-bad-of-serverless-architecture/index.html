Serverless 架构的优劣

你能打开这篇文章，说明你已经在关注 serverless 了，没错，它是云发展的下一个大事件。
你可能会问，我们业务已经上云了，不再拥有任何实体硬件了，这和 serverless 有什么区别？
serverless 的架构下，开发者选中一个语言（Node.js Python C#等）开发对应的服务，上传代码后会被系统自动部署。通过云服务商的生态，你可以描述这些服务之间如何通信，并获取数据。 这是一种接近 NoOps 的方法，因为运维工作外包给了云服务商。

Serverless 101

Function-as-a-Service


Microservices vs serverless

Event-driven coding

如果你家中有安装安防摄像头的话，你不会希望它7*24 小时的运转，而是设定一个动作检测来侦测可疑行为。 serverless 也是类似的机制，只有当预设的事件被触发，它才开始工作。同时 serverless 是无状态的，意味着它只执行任务，不保存或者重用请求。

Scalable services

serverless 架构对于复杂的对可扩展性要求高的应用是理想选择。一个 Fass 提供商可以自动高效无上限的扩容你的函数。

Billing per invocation

serverless 架构是按请求调用实际使用的资源来收费的。

Serverless 优势

  1. 比传统云服务更便宜（按需付费）
  2. 可扩展性更强
  3. 更低的人力成本 （省掉运维人员的成本支出）
  4. 聚焦用户体验 （不再关注服务器的部分）

Serverless 劣势

  1. 绑定服务商。你必须遵守服务商的规则。并且想在不同服务商之间切换可没那么容易
  2. 学习曲线陡峭。从单体应用转向微服务有很多复杂的任务，最好的办法是使用相关的 serverless 工具来自动完成
  3. 不适合长期运行的任务。Lambda 要求任务在5分钟内完成，如果超过需要调用另外一个 function。所以 serverless 适合即时，或者准即时应用。像视频上传这样耗时的任务并不适合

Serverless 适合的 Cases

  1. IoT
  2. 虚拟助手 或 聊天机器人
  3. 富图片应用
  4. 敏捷或者 CI 的 pipeline

Serverless 是未来吗？

有这个势头，预计2-5年内会达到生产率的高峰。现在唯一的问题是扩大适应的范围，增加更多的语言支持，和各服务商扩大自己的函数池！
